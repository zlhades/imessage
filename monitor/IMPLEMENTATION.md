## 🔄 工作流程

```
用户执行：node bin/monitor.js data/incident.jsonl
         ↓
启动监控器 → 创建状态文件
         ↓
开始循环：
  ├─ 等待 30 秒
  ├─ 读取文件新消息
  ├─ 有新消息？
  │   ├─ 是 → 显示 + AI Prompt → 用户复制给 AI
  │   └─ 否 → 继续等待（不消耗 token）
  ├─ 检测用户指令（可随时打断）
  ├─ 检查结束标记？
  │   ├─ 是 → 停止监控
  │   └─ 否 → 继续循环
         ↓
事件结束 → 保存最终状态
```

## 🚀 使用示例

### 场景 1: 本地文件监控

```bash
# 1. 创建消息文件
cat > data/incident.jsonl << 'EOF'
{"ts": "1", "user": "user1", "text": "API 服务返回 500 错误！"}
{"ts": "2", "user": "user2", "text": "我也是"}
EOF

# 2. 启动监控
node bin/monitor.js data/incident.jsonl --interval 5

# 3. 另一个终端添加消息
echo '{"ts": "3", "user": "user1", "text": "错误率 80%"}' >> data/incident.jsonl

# 4. 监控器检测到，输出 AI Prompt
# 5. 复制 Prompt 发送给通义千问/Claude
```

### 场景 2: 单次分析

```bash
# 查看最近消息并生成 AI Prompt
node bin/monitor.js data/incident.jsonl --analyze
```

### 场景 3: 查看状态

```bash
# 查看监控状态
node bin/monitor.js data/incident.jsonl --status
```

## 📋 核心设计决策

### 1. 为什么是命令行工具？

**原因**:
- 简单易用，5 分钟上手
- 可在任何环境运行
- 易于被 AI 调用
- 不需要复杂部署

### 2. 为什么用文件存储状态？

**原因**:
- 跨会话持久化
- 易于调试
- 不依赖外部服务
- 简单可靠

### 3. 如何优化 Token？

**策略**:
- 只在有新消息时调用 AI
- 无新消息时只等待
- 历史消息不重复发送

### 4. 如何支持并行接收？

**实现**:
- 用户指令队列
- 可随时打断主循环
- 优先级：用户输入 > Slack 消息
