- [x] 应该识别问题已解决

**测试结果**:
```
✅ 信息不足时生成追问问题：[
  "请问大概有多少用户受到影响？",
  "具体是哪个服务或 API 端点受影响？",
  "问题是从什么时候开始的？"
]

✅ 需要调查时生成建议：[
  "检查 支付服务 的日志文件",
  "查看 支付服务 的错误率指标",
  "检查最近的代码部署",
  "查看 支付服务 的依赖服务状态"
]

✅ 信息完整时准备回复，状态：READY_TO_REPLY

✅ 识别问题已解决
```

**状态判断规则**:
| 状态 | 触发条件 | 行动 |
|------|---------|------|
| WAITING_INFO | 缺失信息 > 2 项 | 生成追问问题 |
| INVESTIGATING | 缺失信息 1-2 项 | 建议调查方向 |
| READY_TO_REPLY | 信息完整 | 生成回复草稿 |
| READY_TO_CLOSE | 检测到结束标记 | 生成事件总结 |

**测试结果**:
```
=== 完整工作流测试开始 ===

1️⃣  创建测试文件
2️⃣  激活监控
3️⃣  发现新消息：1 条
4️⃣  分析结果：WAITING_INFO
5️⃣  生成追问问题：3 个
6️⃣  用户补充信息
7️⃣  发现新消息：1 条
8️⃣  分析结果：MONITORING

=== 完整工作流测试完成 ===
```

**工作流步骤**:
1. 创建测试文件
2. 激活监控（状态：MONITORING）
3. 检查文件变化
4. 分析消息（状态：WAITING_INFO）
5. 生成追问问题
6. 用户补充信息（写入文件）
7. 再次检查文件变化
8. 再次分析
9. 生成回复草稿（待确认）
10. 用户确认并发送
11. 添加结束标记
12. 检测事件结束
13. 重置状态（状态：IDLE）
