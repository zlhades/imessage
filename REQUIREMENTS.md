## 🤖 AI 快速理解（重要！）

**如果你是 AI 助手，请先阅读本节！**

### 项目一句话总结

**Slack AI Incident Response Bot** = AI 驱动的事件响应机器人 + MCP 架构 + 发送前双重确认 + File/Slack 双模式

### 核心需求（4 个关键点）

1. **双模式设计**: File（开发）/ Slack（生产），切换只需改配置
2. **发送前双重确认**: AI 生成回复 → 用户确认 → 二次 Pull 检查 → 发送
3. **增量更新**: 新消息到达时不取消当前任务，合并处理
4. **AI 行为要求**: 主动追问、系统调查、专业回复

### 核心流程图

```
1. 定时轮询 → 2. 拉取消息 → 3. AI 分析 → 4. 决策
                                              ↓
    ┌─────────────────────────────────────────┘
    ├─ 信息不足 → 生成追问 → 发送
    ├─ 需要调查 → 执行调查 → 重新分析
    └─ 信息完整 → 生成回复 → 双重确认 → 发送
```

### MCP 工具（AI 可调用）

| 工具 | 功能 |
|------|------|
| `file_get_messages` / `slack_get_messages` | 拉取消息 |
| `file_send_message` / `slack_send_message` | 发送消息 |
| `file_check_new_messages` / `slack_check_new_messages` | 检查新消息 |
| `github_get_recent_commits` | 查代码变更 |
| `prometheus_query` | 查指标 |

### 消息格式（File 模式）

```json
{"ts": "1711953600", "user": "user1", "text": "API 返回 500 错误", "thread_ts": "可选"}
```

## 1. 项目概述

### 1.1 项目目标

构建一个 **AI 驱动的事件响应机器人**，实现以下目标：

| 目标 | 说明 | 优先级 |
|------|------|--------|
| 自动监控 | 监控消息源（Slack/文件）中的事件讨论 | P0 |
| 智能分析 | 理解对话内容，识别事件类型和严重性 | P0 |
| 主动追问 | 信息不足时主动询问关键信息 | P1 |
| 系统调查 | 调用工具（GitHub/Prometheus）调查问题 | P1 |
| 安全发送 | 发送前双重确认机制 | P0 |
| 本地交互 | 支持本地补充 Slack 没有的信息 | P2 |

### 1.2 用户故事

```
作为 oncall 工程师，
我希望机器人能自动监控事件频道，
这样我就不用时刻盯着 Slack。

作为事件响应负责人，
我希望机器人能主动追问缺失信息，
这样能更快定位问题。

作为团队领导，
我希望机器人发送回复前有人类确认，
这样避免发送错误或不当的回复。

作为开发者，
我希望在没有 Slack 的环境下也能开发测试，
这样不受网络限制。
```

### 1.3 使用场景

| 场景 | 触发条件 | 预期行为 |
|------|----------|----------|
| 问题报告 | 用户在频道报告问题 | 分析→追问/调查→生成回复 |
| 信息更新 | 用户提供新信息 | 更新上下文→重新分析 |
| 问题升级 | 检测到 P0/P1 事件 | 优先处理，快速响应 |
| 问题解决 | 检测到解决方案 | 记录并确认关闭 |
| 多轮对话 | 多人协作讨论 | 保持上下文连贯性 |

## 2. 核心需求

### 2.1 消息源抽象

**需求描述**: 系统必须支持多种消息源，可热切换。

| 消息源 | 状态 | 说明 |
|--------|------|------|
| File | ✅ 已实现 | 基于 JSONL 文件，用于本地开发 |
| Slack | ✅ 已实现 | 基于 Slack API，用于生产环境 |
| 钉钉 | ⬜ 待实现 | 未来扩展 |
| 飞书 | ⬜ 待实现 | 未来扩展 |

**技术要求**:
- 消息源切换只需修改配置，无需修改代码
- 所有消息源使用统一的 MCP 接口
- 支持同时启用多个消息源（未来）

### 2.2 发送前双重确认

**需求描述**: 机器人发送任何回复前必须经过人类确认。

**流程**:
```
1. AI 生成回复草稿
       ↓
2. 本地显示草稿，等待用户确认
       ↓
3. 用户选择操作：
   ├─ [1] 确认发送 → 执行步骤 4
   ├─ [2] 修改草稿 → 返回步骤 1
   ├─ [3] 进一步调查 → 执行调查
   ├─ [4] 补充本地信息 → 重新分析
   └─ [5] 取消 → 结束
       ↓
4. 第二次 Pull 检查新消息
       ↓
5. 判断是否有新消息：
   ├─ 有新消息 → 重新分析
   └─ 无新消息 → 发送
```

**验收标准**:
- [ ] 所有回复必须经过确认才能发送
- [ ] 确认前必须执行第二次 Pull 检查
- [ ] 检测到新消息时必须重新分析

### 2.3 增量更新策略

**需求描述**: 新消息到达时，默认不取消当前处理中的任务。

| 策略 | 触发条件 | 行为 |
|------|----------|------|
| 增量合并（默认） | 普通新消息 | 加入上下文，等当前任务完成后处理 |
| 立即中断（可选） | P0 紧急事件 | 取消当前任务，重新评估 |

**配置项**:
```yaml
concurrency:
  defaultStrategy: incremental  # incremental / cancel-and-restart
  urgentInterrupt: true
```

## 3. 功能需求

### 3.1 消息拉取

**FR-001**: 系统必须定时从消息源拉取新消息。

| 属性 | 值 |
|------|-----|
| 频率 | 可配置（File 模式：5 秒，Slack 模式：30 秒） |
| 批量 | 可配置（默认 50 条） |
| 过滤 | 自动过滤已处理的消息 |

**FR-002**: 系统必须支持增量拉取（只拉取新消息）。

```typescript
interface PullRequest {
  channel: string;
  limit: number;
  oldest: string;  // 只拉取此时间戳之后的消息
}
```

### 3.2 消息分析

**FR-003**: 系统必须分析每条消息的意图。

**意图分类**:
| 意图 | 说明 | 示例 |
|------|------|------|
| `problem_report` | 问题报告 | "API 返回 500 错误" |
| `information_update` | 信息更新 | "我发现了原因" |
| `question` | 询问 | "有人知道怎么回事吗" |
| `resolution` | 问题解决 | "已修复" |
| `other` | 其他 | 闲聊等 |

**FR-004**: 系统必须识别事件的严重性。

| 级别 | 说明 | 响应要求 |
|------|------|----------|
| P0 | 核心服务完全不可用 | 立即响应 |
| P1 | 大面积影响 | 5 分钟内响应 |
| P2 | 部分影响 | 15 分钟内响应 |
| P3 | 轻微影响 | 30 分钟内响应 |

**FR-005**: 系统必须检测信息完整性。

**检查项**:
- [ ] 受影响的服务/功能
- [ ] 问题开始时间
- [ ] 影响范围（用户数、错误率等）
- [ ] 错误现象/日志
- [ ] 最近的变更

### 3.3 主动追问

**FR-006**: 当信息不完整时，系统必须生成追问问题。

**要求**:
- 问题必须具体、可回答
- 优先询问最关键的信息
- 一次最多 3 个问题

**示例**:
```
缺失信息：["影响范围", "错误率", "开始时间"]

生成的问题：
1. "请问大概有多少用户受到影响？"
2. "错误率大概是多少？"
3. "问题是从什么时候开始的？"
```

### 3.4 调查引擎

**FR-007**: 系统必须支持调用工具进行调查。

**支持的工具**:
| 工具 | 功能 | 状态 |
|------|------|------|
| GitHub | 查代码变更、提交、PR | ✅ 已实现 |
| Prometheus | 查指标、告警、健康状态 | ✅ 已实现 |
| Logs | 查错误日志 | ⬜ 预留 |

**FR-008**: 调查计划必须由 AI 生成并执行。

```typescript
interface InvestigationPlan {
  steps: Array<{
    tool: string;      // 工具名称
    action: string;    // 操作类型
    query: string;     // 查询参数
    purpose: string;   // 调查目的
  }>;
  priority: 'high' | 'medium' | 'low';
}
```

### 3.5 回复生成

**FR-009**: 系统必须生成专业的回复。

**回复要求**:
- 语气专业、冷静
- 包含具体的建议步骤
- 如果不确定，明确说明
- 避免过度承诺

**FR-010**: 回复必须基于最新的上下文。

### 3.6 本地交互

**FR-011**: 系统必须支持本地补充信息。

**输入类型**:
| 类型 | 说明 | 示例 |
|------|------|------|
| `SUPPLEMENT` | 补充 Slack 没有的信息 | "我刚查了日志，发现..." |
| `INSTRUCTION` | 指导 AI | "去查过去 1 小时的错误" |
| `CORRECTION` | 纠正 AI | "不是数据库问题，是网络问题" |

**FR-012**: 本地输入必须合并到上下文中。

### 3.7 状态持久化

**FR-013**: 系统必须持久化运行时状态。

**持久化内容**:
- 已处理消息 ID（避免重复处理）
- 最后拉取时间戳
- 待发送草稿（可选）
- 本地输入历史（可选）

**存储方式**: JSON 文件（可扩展到 Redis）

```
│                        │ 调用 MCP 工具                   │
│                        ▼                                │
│  ┌────────────────────────────────────────────────────┐ │
│  │                 MCP Servers                         │ │
│  │  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌───────┐ │ │
│  │  │   File   │ │  Slack   │ │ GitHub   │ │  ...  │ │ │
│  │  │ (开发)   │ │ (生产)   │ │ (调查)   │ │       │ │ │
│  │  └──────────┘ └──────────┘ └──────────┘ └───────┘ │ │
│  └────────────────────────────────────────────────────┘ │

```

### 5.2 模块依赖

```
src/index.ts (主入口)
    │
    ├─→ src/mcp/client.ts (MCP 客户端)
    │       └─→ 连接所有启用的 MCP Servers
    │
    ├─→ src/ai/provider.ts (AI Provider 抽象)
    │       ├─→ src/ai/qwen-provider.ts
    │       └─→ src/ai/claude-provider.ts
    │
    ├─→ src/analysis/ (分析模块)
    │       ├─→ intent.ts (意图识别)
    │       └─→ completeness.ts (完整性检查)
    │
    ├─→ src/investigation/engine.ts (调查引擎)
    │
    ├─→ src/approval/gateway.ts (发送前确认)
    │
    ├─→ src/conversation/merge.ts (上下文合并)
    │
    └─→ src/cli/interactive.ts (本地交互)
```

### 5.3 数据流

```
1. 定时触发
       ↓
2. MCP: pull_messages()
       ↓
3. 过滤已处理消息
       ↓
4. AI: analyze(context)
       ↓
5. 决策：
   ├─ 信息不足 → generateQuestions() → 发送追问
   ├─ 需要调查 → executeInvestigation() → 重新分析
   └─ 信息完整 → generateResponse()
       ↓
6. Approval Gateway:
   ├─ 准备草稿
   ├─ 用户确认
   ├─ 二次 Pull 检查
   └─ 发送
       ↓
7. 更新状态（持久化）
```

## 6. File 模式需求

### 6.1 核心需求

**FR-F001**: File 模式必须模拟 Slack 的所有核心功能。

| Slack 功能 | File 模拟 | 实现状态 |
|-----------|----------|----------|
| 拉取消息 | 读取 JSONL 文件 | ✅ |

| 检查新消息 | 时间戳比较 | ✅ |
| 线程回复 | thread_ts 字段 | ✅ |

**FR-F002**: File 模式必须支持命令行工具。


```json
{"ts": "1711953600", "user": "user1", "text": "消息内容", "thread_ts": "可选"}
```

| 检测方式 | 时间戳比较 |

### 6.4 并发处理

**FR-F006**: File 模式必须处理并发写入。

**当前实现**: 简单追加（适合单用户）
**未来改进**: 文件锁机制

### 6.5 测试需求

**FR-F007**: File 模式必须有完整的 E2E 测试。

**测试覆盖**:

## 8. AI 行为需求

### 8.1 意图识别

**FR-A001**: AI 必须准确识别消息意图。

**准确率要求**:
| 意图类型 | 最低准确率 |
|---------|-----------|
| problem_report | > 90% |
| information_update | > 85% |
| question | > 90% |
| resolution | > 85% |

### 8.2 严重性判断

**FR-A002**: AI 必须根据关键词判断严重性。

**判断规则**:
| 关键词/模式 | 严重性 |
|------------|--------|
| "down", "完全不可用", "P0" | P0 |
| "大面积", "核心功能" | P1 |
| "部分", "某些用户" | P2 |
| "偶尔", "轻微" | P3 |

### 8.3 追问行为

**FR-A003**: AI 必须在信息不足时主动追问。

**追问原则**:
1. 优先询问最关键的信息
2. 问题具体、可回答
3. 一次最多 3 个问题
4. 避免重复询问

### 8.4 调查行为

**FR-A004**: AI 必须系统性调查问题。

**调查顺序**:
1. 检查服务健康状态（Prometheus）
2. 查看最近的代码变更（GitHub）
3. 查询错误日志（Logs）
4. 对比历史数据

### 8.5 回复风格

**FR-A005**: AI 回复必须符合专业标准。

**风格要求**:
- ✅ 专业、冷静
- ✅ 具体、可操作
- ✅ 承认不确定性
- ❌ 避免过度承诺
- ❌ 避免模糊表述

**示例**:
```
✅ "根据分析，错误率上升与 14:25 的部署相关。
    建议：1) 回滚部署 2) 检查配置变更"

❌ "可能是部署的问题，你看看吧"
```
