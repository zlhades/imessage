## 1. 项目概述

### 1.1 项目背景

**Typeless** 是一款 AI 驱动的语音输入软件，用户可以通过自然语言说话，自动生成清晰、专业、格式良好的文本。它支持 macOS、Windows、iOS、Android 四端，核心亮点包括：

- 🎤 **智能语音转文字**：实时转写，速度约 220 词/分钟
- 🧹 **自动清理**：去除语气词、重复、自我纠正
- 📝 **自动格式化**：列表、步骤、要点自动结构化
- 🎨 **个性化风格**：学习用户语气和习惯用语
- 🌍 **实时翻译**：支持 100+ 语言互译
- 🔒 **隐私保护**：零云端数据保留

### 1.2 项目目标

开发一款与 Typeless 功能相近的 **AI 智能语音输入软件**，实现以下目标：

| 目标 | 说明 | 优先级 |
|------|------|--------|
| 语音输入 | 说话即打字，4 倍于键盘输入速度 | P0 |
| 智能处理 | 自动清理、格式化、风格化 | P0 |
| 全局可用 | 系统级热键触发，任意应用可用 | P0 |
| 隐私优先 | 本地处理，不上传用户数据 | P1 |
| 多语言 | 支持中英文及主流语言互译 | P1 |
| 跨平台 | macOS + Windows + iOS + Android | P2 |

### 1.3 用户故事

```
作为文字工作者，
我希望通过语音快速生成文档，
这样不用长时间打字导致疲劳。

作为程序员，
我希望语音描述能自动格式化为代码注释，
这样写文档效率更高。

作为多语言用户，
我希望说话后自动翻译成目标语言，
这样跨语言沟通无障碍。

作为注重隐私的用户，
我希望所有数据仅在本地处理，
这样不用担心数据泄露。
```

```
│ 用户说话 │────►│ 音频采集  │────►│ 语音识别  │────►│ AI 处理  │────►│ 文本注入 │

                     ▼                  ▼                  ▼
```

### 4.2 详细处理流程

```
1. 触发录音
   │
   ├─ 用户按下全局热键 (如 ⌘ + Space)
   ├─ 系统开始采集麦克风音频
   └─ VAD (Voice Activity Detection) 检测语音开始
         │
         ▼
2. 音频预处理
   │
   ├─ 降噪处理 (WebRTC AudioProcessing)
   ├─ 回声消除 (AEC)
   ├─ 自动增益控制 (AGC)
   └─ 分帧处理 (每 20ms 一帧)
         │
         ▼
3. 语音识别 (ASR)
   │
   ├─ 流式 Whisper 推理
   ├─ 实时输出文字片段 (每 500ms 更新)
   ├─ VAD 检测到静音 > 1s → 结束录音
   └─ 输出原始文本 (含语气词、重复等)
         │
         ▼
4. AI 文本处理
   │
   ├─ 规则引擎：去除语气词 ("嗯"、"啊"、"那个")
   ├─ 规则引擎：去除重复词句
   ├─ 规则引擎：处理自我纠正 ("不对，应该是...")
   ├─ LLM 重构：智能标点、大小写、段落划分
   ├─ LLM 格式化：识别列表、步骤、要点并结构化
   └─ 风格匹配：根据当前应用调整语气
         │
         ▼
5. 文本注入
   │
   ├─ 检测当前活跃应用
   ├─ 查询应用类型 (邮件/聊天/代码编辑器)
   ├─ 调整输出格式
   └─ 通过 Accessibility API 注入文本
         │
         ▼
6. 完成
   │
   ├─ 记录到本地历史
   ├─ 更新用户画像 (可选)
   └─ 等待下次触发
```

## 6. 核心模块详细设计

### 6.1 语音识别模块 (ASR)

**职责**: 将用户语音实时转为文字

#### 技术方案

```
音频流 → VAD 检测 → Whisper 流式推理 → 文本输出
```

#### 关键参数

| 参数 | 值 | 说明 |
|------|-----|------|
| 采样率 | 16kHz | Whisper 要求 |
| 位深度 | 16-bit | 标准音频质量 |
| 帧大小 | 20ms | 流式处理粒度 |
| 更新频率 | 500ms | 文本输出刷新率 |
| 静音阈值 | 1s | 超过此时间自动结束录音 |

#### 模型选型

| 模型 | 大小 | 速度 (RTF) | 准确率 (WER) | 适用场景 |
|------|------|-----------|-------------|----------|
| Whisper Tiny | 39MB | 0.15 | 8.5% | 移动端、低功耗 |
| Whisper Base | 74MB | 0.25 | 7.2% | 移动端 |
| Whisper Small | 244MB | 0.50 | 5.8% | 桌面端 |
| Whisper Medium | 769MB | 1.00 | 4.9% | 桌面端 |
| Whisper Large-v3 | 1.5GB | 2.00 | 4.2% | 高精度场景 |

> **RTF** = Real Time Factor, 1.0 表示处理 1 秒音频需要 1 秒计算时间  
> **WER** = Word Error Rate, 越低越好

#### 开发难点

- ⚠️ **流式识别**: Whisper 原生不支持真正的流式识别，需要使用 `faster-whisper` 或自研方案
- ⚠️ **噪音环境**: 需要 WebRTC 降噪 + 自适应阈值
- ⚠️ **移动端性能**: 需要使用 CoreML (iOS) / NNAPI (Android) 加速

### 6.2 AI 文本处理引擎

**职责**: 对 ASR 输出进行智能清理、格式化、风格化

#### 处理流程

```
原始 ASR 输出
    │
    ├── 规则引擎处理
    │    ├── 去除语气词 ("嗯"、"啊"、"那个"、"like"、"um")
    │    ├── 去除重复 ("我觉得...我觉得" → "我觉得")
    │    └── 处理自我纠正 ("不对，应该是..." → "应该是...")
    │
    ├── LLM 文本重构
    │    ├── 智能标点 (根据语义添加句号、逗号)
    │    ├── 大小写修正 (英文首字母大写、专有名词)
    │    └── 段落划分 (根据话题转换分段)
    │
    └── 风格化输出
         ├── 应用上下文检测
         ├── 语气调整 (正式/轻松/技术)
         └── 结构化格式化 (列表、步骤、要点)
```

#### LLM Prompt 示例

```yaml
# 系统提示词模板
system_prompt: |
  你是一个专业的文本处理助手。请对以下语音识别输出进行处理：

  要求：
  1. 添加正确的标点符号和大小写
  2. 去除语气词和重复内容
  3. 保持用户的原始意思不变
  4. 根据应用类型调整语气：
     - 邮件/文档：正式、完整句式
     - 聊天/社交：轻松、短句
     - 代码编辑器：技术术语、英文优先
  5. 如果用户说了列表或步骤，请格式化为清晰的结构

  输入文本: {asr_output}
  当前应用: {app_name}
  应用类型: {app_category}

  请输出处理后的文本，不要添加额外解释。
```

#### 开发难点

- ⚠️ **延迟控制**: LLM 推理需要 < 1 秒，需模型量化优化
- ⚠️ **流式输出**: 避免整句重算导致的文本闪烁
- ⚠️ **个性化**: 需要积累用户数据才能有效风格匹配

### 6.3 文本注入模块

**职责**: 将处理后的文本注入到任意应用中

#### 各平台实现

| 平台 | 技术方案 | 兼容性 | 限制 |
|------|----------|--------|------|
| **macOS** | Accessibility API (`AXUIElementSetAttributeValue`) | 90%+ | 需要辅助功能权限 |
| **Windows** | `SendInput()` API / UI Automation | 85%+ | 部分应用阻止模拟输入 |
| **iOS** | 自定义键盘扩展 (`UIInputViewController`) | 100% | 需要用户切换键盘 |
| **Android** | `InputMethodService` | 100% | 需要用户切换输入法 |

#### macOS 注入示例

```rust
// 伪代码：通过 Accessibility API 注入文本
fn inject_text(text: &str) -> Result<(), Error> {
    let app = get_foreground_app()?;
    let ui_element = app.focused_ui_element()?;
    
    // 模拟键盘输入
    for char in text.chars() {
        simulate_key_event(char);
        sleep(Duration::from_millis(5)); // 避免输入过快
    }
    
    Ok(())
}
```

#### 开发难点

- ⚠️ **权限申请**: macOS 需要辅助功能权限，Windows 可能需要管理员权限
- ⚠️ **兼容性问题**: 某些应用（如终端、密码框）不允许注入
- ⚠️ **移动端限制**: iOS 键盘扩展功能受限，审核严格

### 6.4 上下文感知模块

**职责**: 检测当前活跃应用，自动调整输出语气和格式

#### 应用分类字典 (示例)

```yaml
categories:
  email:
    apps: ["Mail", "Outlook", "Gmail", "Spark"]
    tone: formal
    format: complete_sentences

  chat:
    apps: ["微信", "Slack", "Discord", "Telegram", "WhatsApp"]
    tone: casual
    format: short_sentences

  code:
    apps: ["VS Code", "Xcode", "IntelliJ", "Vim"]
    tone: technical
    format: mixed

  document:
    apps: ["Word", "Pages", "Google Docs", "Notion"]
    tone: formal
    format: structured

  social:
    apps: ["Twitter", "微博", "小红书", "Instagram"]
    tone: casual
    format: short
```

#### 开发难点

- ⚠️ **API 差异**: macOS 和 Windows 的前台应用检测 API 不同
- ⚠️ **字典维护**: 需要持续更新应用分类字典
- ⚠️ **用户自定义**: 需要允许用户自定义规则

### 6.5 隐私与数据模块

**职责**: 确保用户数据安全，实现零云端存储

#### 本地存储结构

```
SQLite 数据库 (typeless.db)
├── history
│    ├── id (INTEGER PRIMARY KEY)
│    ├── timestamp (INTEGER)
```

## 7. 模型接入方案

### 7.1 模型总览

本软件需要接入 **三类 AI 模型**：

| 模型类型 | 用途 | 推荐方案 | 部署方式 |
|----------|------|----------|----------|
| **ASR 模型** | 语音转文字 | OpenAI Whisper | 本地运行 |
| **LLM 模型** | 文本处理、格式化、风格化 | Qwen / Llama 3 | 本地或云端 |
| **翻译模型** | 多语言互译 | NLLB-200 / 云端翻译 API | 本地或云端 |

### 7.2 ASR 模型 - OpenAI Whisper

#### 方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **faster-whisper** (CTranslate2) | 速度 4x、内存减半 | 需要 C++ 依赖 | 桌面端 (推荐) |
| **whisper.cpp** | 纯 C++、跨平台、CoreML 支持 | 准确率略低 | 桌面端 + 移动端 |
| **OpenAI API** | 零部署成本、最新版模型 | 按量付费、延迟高 | MVP 快速验证 |
| **自研流式方案** | 真正的流式识别 | 开发成本高 | 长期方案 |

#### 推荐接入方式

**MVP 阶段**: 使用 OpenAI API 快速验证
```python
import openai

# 使用 OpenAI Whisper API
audio_file = open("recording.wav", "rb")
transcript = openai.Audio.transcribe(
    model="whisper-1",
    file=audio_file,
    language="zh"  # 可选：指定语言提高准确率
)
```

**v1.0 阶段**: 使用 faster-whisper 本地部署
```python
from faster_whisper import WhisperModel

# 加载模型 (桌面端使用 Medium 模型)
model = WhisperModel("medium", device="cpu", compute_type="int8")

# 流式推理
segments, info = model.transcribe(
    "audio.wav",
    beam_size=5,
    vad_filter=True,  # 内置 VAD
    language="zh"
)

for segment in segments:
    print(f"[{segment.start:.2f}s -> {segment.end:.2f}s] {segment.text}")
```

#### 硬件要求

| 模型 | CPU 推理 | GPU 推理 | 内存需求 |
|------|----------|----------|----------|
| Tiny | ✅ 流畅 | - | 512MB |
| Base | ✅ 流畅 | - | 1GB |
| Small | ✅ 可用 | ✅ 快速 | 2GB |
| Medium | ⚠️ 较慢 | ✅ 流畅 | 4GB |
| Large-v3 | ❌ 太慢 | ✅ 流畅 (8GB VRAM) | 8GB |

### 7.3 LLM 模型 - 文本处理引擎

#### 方案对比

| 方案 | 模型 | 优点 | 缺点 | 成本 |
|------|------|------|------|------|
| **本地部署** | Qwen 7B / Llama 3 8B | 隐私好、无 API 费 | 需 8GB+ RAM | 硬件成本 |
| **云端 API** | GPT-4o-mini | 质量好、零部署 | 按 token 付费、隐私风险 | $0.15/1M tokens |
| **云端 API** | Qwen API (DashScope) | 中文好、价格低 | 按 token 付费 | ¥0.008/1K tokens |
| **混合方案** | 本地小模型 + 云端大模型 | 平衡 | 架构复杂 | 中等 |

#### 推荐接入方式

**方案 A: 纯本地 (推荐，注重隐私)**

```python
# 使用 llama-cpp-python 加载量化模型
from llama_cpp import Llama

llm = Llama(
    model_path="./qwen-7b-q4_k_m.gguf",  # 4-bit 量化
    n_ctx=2048,        # 上下文窗口
    n_threads=8,       # CPU 线程数
    n_gpu_layers=32    # GPU 加速层数 (可选)
)

# 文本处理
response = llm(
    prompt=system_prompt + user_input,
    max_tokens=512,
    temperature=0.3,   # 低温度保证稳定性
    stop=["\n\n"]      # 停止标记
)

print(response["choices"][0]["text"])
```

**硬件要求**:
- CPU 推理: 8GB RAM (4-bit 量化 7B 模型约 4GB)
- GPU 推理: 6GB VRAM (NVIDIA / Apple Silicon)
- 延迟: 1-3 秒 (CPU), 0.5-1 秒 (GPU)

**方案 B: 云端 API (快速启动，成本低)**

```python
# 使用 DashScope (Qwen API)
import dashscope

dashscope.api_key = "your-api-key"

response = dashscope.Generation.call(
    model="qwen-turbo",
    prompt=system_prompt + user_input,
    max_tokens=512,
    temperature=0.3
)

print(response.output.text)
```

**成本预估** (按每天 1000 次调用，每次 200 token):
- GPT-4o-mini: $0.15/1M tokens → **约 $4.5/月**
- Qwen Turbo: ¥0.008/1K tokens → **约 ¥2.4/月 (极便宜)**

| **DeepL API** | 质量最好、延迟低 | 付费、€5.49/月 | 商业版 |

#### 推荐接入方式

**MVP 阶段**: 使用 Qwen 多语言能力
```
用户输入: "你好，我想去北京"
Prompt: "Translate the following text to English: {input}"
输出: "Hello, I want to go to Beijing"
```

**v1.0 阶段**: 使用 NLLB-200 本地部署
```python
from transformers import AutoModelForSeq2SeqLM, AutoTokenizer

model = AutoModelForSeq2SeqLM.from_pretrained("facebook/nllb-200-distilled-600M")
tokenizer = AutoTokenizer.from_pretrained("facebook/nllb-200-distilled-600M")

# 中英翻译
inputs = tokenizer("你好世界", src_lang="zho_Hans", return_tensors="pt")
outputs = model.generate(**inputs, forced_bos_token_id=tokenizer.lang_code_to_id["eng_Latn"])
print(tokenizer.decode(outputs[0], skip_special_tokens=True))
# 输出: "Hello World"
```

## 15. 总结与建议

### 15.1 项目可行性评估

| 维度 | 评估 | 说明 |
|------|------|------|
| **技术可行性** | ✅ 可行 | 开源 ASR 和 LLM 已成熟，但有性能挑战 |
| **市场可行性** | ✅ 可行 | 语音输入市场需求增长，Typeless 已验证 |
| **商业可行性** | ⚠️ 需谨慎 | 需明确盈利模式，避免与巨头直接竞争 |
| **团队可行性** | ⚠️ 有挑战 | 需要 AI + 跨平台开发复合团队 |

### 15.2 关键建议

1. **MVP 先行**: 先做 macOS 单平台 MVP，验证核心技术后再扩展
2. **技术储备**: 提前 2-3 个月进行 ASR 和 LLM 方案验证
3. **差异化竞争**: 强调"本地优先"和"隐私保护"，避开巨头优势领域
4. **社区驱动**: 开源核心引擎，建立开发者社区，降低开发成本
5. **敏捷迭代**: 每 2 周一个迭代，快速响应用户反馈
6. **合规先行**: 早期就进行 GDPR 等隐私合规设计
7. **混合架构**: 支持本地 + 云端双模式，用户可自选

### 15.3 关键里程碑检查点

| 里程碑 | 时间 | 检查标准 |
|--------|------|----------|
| ASR 方案确认 | Week 4 | 准确率 > 95%，延迟 < 500ms |
| MVP 可用 | Week 16 | macOS 端语音→文本→注入完整流程可用 |
| 四端基础功能 | Week 24 | 四端均可语音输入 |
